# v0.18.0.0 — Project documents move local + opt-in cross-project archive

## What changed

**Project documents are now fully local to the repo.**

Design docs, test plans, autoplan restore files, CEO plans, checkpoints, branch
reviews, evals, and health history all live in `.nstack/` inside the repo, not
in `~/.nstack/projects/$SLUG/`.

```
.nstack/
├── config.yaml             # local config (cross_project_learnings, etc.)
├── learnings.jsonl         # live learnings (always local)
├── timeline.jsonl          # session timeline
├── designs/                # design docs (from /office-hours, /plan-* skills)
├── plans/                  # test plans, autoplan restore, CEO handoffs
├── plans/ceo/              # CEO plans
├── checkpoints/            # checkpoint files (from /checkpoint)
├── evals/                  # eval results
└── $BRANCH-reviews.jsonl   # branch review log
```

This corrects a v0.17.0.0 partial-ship: the v0.17 CHANGELOG claimed "design docs
and checkpoints now live in `.nstack/`," but the SKILL.md.tmpl files were not
updated, so templates kept writing to the legacy global path. v0.18 finishes
that migration.

**Cross-project learning archive added.**

When `cross_project_learnings=true` (local config, **default `false`**), each
learning logged to `.nstack/learnings.jsonl` is also dual-written to a durable
global archive at `~/.nstack/learnings.jsonl`. This archive survives repo
deletion and is the data source for cross-project search (`/learn`,
`nstack-learnings-search --cross-project`). The global archive is scaffolding
for a future knowledge-graph feature.

Each global entry carries `project` (slug) and `repo_path` metadata so entries
remain attributable after the originating repo is gone.

**Project registry added.**

`~/.nstack/projects.yaml` is a path-keyed map of opted-in projects on this
machine. It enables cross-project operations without re-solving discovery each
time. **Path-keyed (not slug-keyed)** so multiple Conductor worktrees of the
same repo each get their own entry. Opted-out repos are not registered at all.

```yaml
projects:
  "/Users/you/code/repo-A":
    slug: youname-repoA
    added: 2026-04-28T16:13:43Z
    last_seen: 2026-04-28T16:13:43Z
    archived: false
```

**`cross_project_learnings` defaults to `false` (privacy-first).**

Previously the value was "unset" with an interactive AskUserQuestion fallback.
v0.18 sets it explicitly to `false` for new and migrating users. To opt in,
either answer the per-repo prompt on first use, or run:

```bash
nstack-config set --local cross_project_learnings true
```

Setting it back to `false` automatically removes the repo from the registry
(privacy boundary enforced by config change, no separate cleanup step required).

## CLI additions

### `nstack-registry`

| Subcommand | What it does |
|---|---|
| `nstack-registry add` | Register the current cwd's repo (idempotent; no-op if opted out) |
| `nstack-registry list` | Print registered projects (TSV) |
| `nstack-registry status` | Show this repo's registration + opt-in state |
| `nstack-registry prune [--dry-run]` | Mark missing paths as archived |
| `nstack-registry remove --local` | Remove the current cwd entry |
| `nstack-registry remove --path <p>` | Remove a specific path |
| `nstack-registry remove --global` | Remove ALL entries for the current slug |

CLI flags mirror `nstack-config`: `--local` for the current repo, `--global` for
machine-wide.

### `nstack-learnings-log` size limits

- `insight` ≤ 4096 bytes (rejects oversize entries with stderr error)
- `files` array ≤ 16 entries

Defense against output-injection blast: large strings in `insight` and lots of
file paths in `files` get rendered back to terminals during cross-project
summaries, so unbounded inputs are now hard-rejected.

## Migration

Migration runs automatically on `/nstack-upgrade`. Manual: `bash
nstack-upgrade/migrations/v0.18.0.0.sh`.

The migration:

1. Backs up `~/.nstack/projects/` to `~/.nstack/projects.v0.17-backup/`.
2. For each `~/.nstack/projects/$SLUG/`, scans the slug-cache to build a
   slug → [paths] reverse map. The most-recently-touched path is canonical.
   Multi-worktree case (same slug, multiple paths): picks one canonical
   target and prints a notice.
3. Moves design docs, plans, ceo-plans, checkpoints, evals, reviews into the
   target repo's `.nstack/` subdirs.
4. Seeds learnings into `~/.nstack/learnings.jsonl` tagged with
   `project` + `repo_path` (one-time bulk seed).
5. Writes a manifest at `~/.nstack/v0.18-migration-manifest.jsonl` recording
   each move so subsequent runs skip already-moved files.
6. Sets `cross_project_learnings=false` globally if unset (privacy default).
7. Marks complete via `~/.nstack/.v0.18-migrated`.

Migration is **idempotent** (re-running is safe), **non-fatal** (failures log
but don't block upgrade), and **concurrent-safe** (holds a flock so two
`/nstack-upgrade` runs from different worktrees serialize).

## Edge cases handled

| Scenario | Behavior |
|---|---|
| Project-scoped agent (no `$HOME` access) | Local writes succeed; global write skipped with stderr warning |
| Concurrent learnings writes from N worktrees | Lock-protected via `scripts/internal/lock.sh` |
| Multi-worktree same slug | Registry path-keyed; each worktree registers independently |
| Symlinked repos (Conductor) | `realpath -e` resolves before persistence; no double-register |
| Concurrent migration from 2 sessions | Flock on `~/.nstack/.migration-v0.18.lock`; second waits |
| Orphaned legacy data (no slug-cache entry) | Left in place; migration prints orphan note |
| Multi-path slug | Canonical = most-recent slug-cache mtime; others printed as worktrees |
| Malformed JSON in shared file | Reader emits stderr warning, skips line |
| YAML registry corruption | Atomic write; manual edit caution noted in file header |

## Rollback

**Forward-only migration.** The v0.18 → v0.17 downgrade does NOT recover
post-migration writes — any learnings, designs, or plans logged after migration
exist only at the new local paths and are invisible to v0.17 binaries.

If you must roll back:

1. Restore `~/.nstack/projects/` from `~/.nstack/projects.v0.17-backup/`.
2. Downgrade nstack to v0.17.0.0.
3. Accept that work logged in v0.18 (after migration) will not be readable.

## Configuration reference

Local-only (per-repo, in `.nstack/config.yaml`):

| Key | Default | Purpose |
|---|---|---|
| `cross_project_learnings` | `false` | Dual-write learnings to global archive + register the repo |

Global (in `~/.nstack/config.yaml`):

| Key | Default | Purpose |
|---|---|---|
| `cross_project_learnings` | `false` | Fallback when local is unset |

Setting `--local cross_project_learnings false` triggers an automatic
`nstack-registry remove --local` to keep the registry clean.
